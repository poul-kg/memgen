# MemGen v2 - YAML Storage Migration

## Overview

This spec documents the migration from Markdown (`.md`) knowledge files generated by the Claude CLI (LLM) to structured YAML (`.yaml`) files assembled directly by Go code. This was implemented in commit `073973e` and eliminates the `claude` CLI as a runtime dependency.

## Key Changes

- **Storage format**: `.md` -> `.yaml` with structured Go types (`internal/knowledge/types.go`)
- **Claude CLI removed**: Entire `internal/claude/` package deleted. Knowledge is assembled by converting source data directly into Go structs and marshaling to YAML via `gopkg.in/yaml.v3`.
- **Deterministic output**: No LLM summarization; raw data is stored as-is in structured form.
- **Startup validation**: Only `gh` CLI is validated at startup. Claude CLI check removed from `main.go`.
- **Deps struct**: `Claude` field removed from `tools.Deps`. Only `Store`, `Locks`, `JIRA`, `GitHubExecutor`, and `JIRABaseURL` remain.

## YAML Structure

Defined in `internal/knowledge/types.go`. The root type is `KnowledgeFile`:

```yaml
ticket_id: STITCH-1234
branch: feature/STITCH-1234-password-reset
last_refreshed: 2026-04-10T12:00:00Z
jira:
  summary: "Implement password reset flow"
  description: "Full description..."
  status: "In Progress"
  priority: "High"
  assignee: "Alice Chen"
  reporter: "Bob Miller"
  labels: [auth, security]
  comments:
    - author: "Carol N"
      created: 2026-04-03T10:00:00Z
      updated: 2026-04-03T10:00:00Z
      body: "Comment text"
pull_requests:
  - number: 142
    title: "STITCH-1234: Implement password reset"
    state: MERGED
    author: alicec
    url: https://github.com/org/repo/pull/142
    created_at: 2026-04-03T10:00:00Z
    updated_at: 2026-04-06T10:00:00Z
    branch: feature/STITCH-1234-password-reset
    body: "PR description"
    reviews:
      - author: bobm
        state: APPROVED
        body: "Looks good"
        created_at: 2026-04-05T10:00:00Z
    comments:
      - id: 90001
        author: bobm
        body: "Token expiry should be configurable"
        path: internal/auth/token.go
        created_at: 2026-04-04T10:05:00Z
        updated_at: 2026-04-04T10:05:00Z
        in_reply_to: 0
        resolved: true
    commits:
      - sha: abc123
        message: "STITCH-1234: initial implementation"
        author: alicec
        date: 2026-04-03T15:00:00Z
main_commits:
  - sha: def456
    message: "STITCH-1234: merged to main"
    author: alicec
    date: 2026-04-07T10:00:00Z
notes:
  - date: 2026-04-08T14:00:00Z
    body: "Decided to use bcrypt cost factor 12 in next security review"
```

All slices are initialized to non-nil empty slices (both in `FromSources()` and after `yaml.Unmarshal`) to produce clean YAML output without `null` values.

## Source-to-YAML Conversion

Implemented in `internal/knowledge/convert.go`:

- `FromSources(ticketID, branch, jira, prs, mainCommits)` -- creates a complete `KnowledgeFile` from source types.
- `convertJIRA(jira)` -- maps `sources.JIRATicket` to `JIRASection`. Handles nil JIRA gracefully.
- `convertPRs(prs)` / `convertPR(pr)` -- maps `sources.PR` to `PullRequest`, including nested reviews, comments, and commits.
- `convertCommits(commits)` -- maps `sources.Commit` to `CommitEntry`.

## Updated Tool Signatures

| Tool | Args | Changes |
|------|------|---------|
| `memgen__get` | `branch`, `scope` (optional) | Added `scope` param with enum: `"jira"`, `"pr"`, `"git"`, `"comments"`, `"notes"`. Empty = full file. |

## PR Filtering (`filterPREntriesByTicket`)

Implemented in `internal/sources/github.go`. GitHub's `gh pr list --search` uses fuzzy matching which can return false positives (e.g., searching for "SBUX-404" might match unrelated PRs containing "404").

The filter checks for an exact case-insensitive match of the ticket ID in:
- PR title
- PR body
- PR branch name (`headRefName`)

Only PRs matching at least one of these are kept. This runs after the `gh pr list` call and before fetching per-PR details (reviews, comments, commits).

## PR Comment Resolved Status (GraphQL)

Implemented in `internal/sources/github.go` via `fetchReviewThreads()`.

**How it works**:
1. For each PR, after fetching review comments via REST API, a GraphQL query is sent:
   ```graphql
   query {
     repository(owner: "...", name: "...") {
       pullRequest(number: N) {
         reviewThreads(first: 100) {
           nodes {
             isResolved
             comments(first: 1) {
               nodes { databaseId }
             }
           }
         }
       }
     }
   }
   ```
2. The response maps each review thread's first comment `databaseId` to its `isResolved` status.
3. This map is used to set the `Resolved` field on each `PRComment`.
4. **Graceful fallback**: If the GraphQL call fails (e.g., token lacks `read:discussion` scope), all comments default to `resolved: false` and a warning is logged. No error is returned.

The `PRComment` struct gained two new fields: `ID` (GitHub database ID, for correlation with GraphQL data) and `Resolved` (bool).

## Scoped Reads (`ReadSection`)

Implemented in `internal/knowledge/store.go`. The `ReadSection(repo, ticketID, scope)` method supports:

| Scope | Returns |
|-------|---------|
| `""` (empty) | Raw file content as string (same as `Read()`) |
| `"jira"` | `kf.JIRA` marshaled to YAML |
| `"pr"` | Slim PR list: only `number`, `title`, `state`, `author`, `url`, `body` (no reviews/comments/commits) |
| `"git"` | Struct with `pull_requests` (full) and `main_commits` |
| `"comments"` | Aggregated `pr_comments` and `pr_reviews` across all PRs |
| `"notes"` | `kf.Notes` marshaled to YAML |

Unknown scopes return an error listing valid options.

## Docker Setup

### Dockerfile

Multi-stage build in project root:
1. **Builder**: `golang:latest`, copies source, runs `CGO_ENABLED=0 go build -o memgen ./cmd/memgen`.
2. **Runtime**: `debian:bookworm-slim` with `ca-certificates`, `curl`, `git`, and `gh` CLI installed from GitHub's apt repository.
3. Entrypoint script that sets up `PATH` and runs `memgen`.

### docker-compose.yml

- Runs as the host user (`UID`/`GID` env vars, default 1000).
- Exposes port 3040.
- Volume mounts (from host):
  - `~/.config/gh` (read-only) -- `gh` CLI authentication.
  - `~/.config/memgen` -- MemGen config and knowledge storage.
  - `~/.local/share/claude` (read-only) -- Claude CLI versions (legacy, for entrypoint compatibility).
  - `~/.claude` and `~/.claude.json` (read-only) -- Claude auth/config (legacy, for entrypoint compatibility).
- Sets `HOME=/home/memgen` in the container.

## Legacy `.md` File Handling

The store detects legacy `.md` files. If `ReadKnowledge()` is called and the `.yaml` file doesn't exist but a `.md` file does, it returns a descriptive error: "knowledge file <TICKET-ID> has legacy .md format; please re-run init to convert to YAML."
